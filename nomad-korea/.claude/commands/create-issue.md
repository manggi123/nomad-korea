당신은 GitHub 이슈를 생성하는 전문가입니다.

## 입력된 내용
```
$ARGUMENT
```

## 작업 단계

1. **이슈 내용 분석**
   - 입력된 내용에서 제목과 본문을 파악
   - 만약 제목과 본문이 명확히 구분되지 않으면 사용자에게 질문

2. **이슈 구조화**
   - 제목: 명확하고 간결하게 (50자 이하 권장)
   - 본문: 다음 섹션을 포함하여 구조화
     - **📋 개요**: 이슈의 목적과 배경
     - **✅ 작업 내용**: 구체적인 작업 항목 (체크박스 사용)
     - **🎯 완료 기준**: 언제 이슈가 완료되는지 명확히 정의
     - **📝 참고 사항**: 추가 정보나 링크 (필요시)

3. **레이블 추천**
   - 이슈 내용을 분석하여 적절한 레이블 추천
   - 예: `enhancement`, `bug`, `documentation`, `question`, `help wanted`

4. **우선순위 판단**
   - P0 (긴급), P1 (높음), P2 (보통), P3 (낮음)

5. **GitHub CLI로 이슈 생성**
   - `gh issue create` 명령어 사용
   - HEREDOC을 사용하여 본문 작성
   - 레이블과 우선순위 추가

## 예시

입력: "도시 검색 결과가 없을 때 빈 화면만 표시됨"

출력:
```bash
gh issue create \
  --title "검색 결과 없음 시 안내 메시지 추가" \
  --label "enhancement,ux" \
  --body "$(cat <<'EOF'
## 📋 개요
현재 도시 검색 결과가 없을 때 빈 화면만 표시되어 사용자가 혼란스러워할 수 있습니다. 명확한 안내 메시지를 추가하여 UX를 개선합니다.

## ✅ 작업 내용
- [ ] "검색 결과가 없습니다" 메시지 추가
- [ ] 검색어 수정 제안 표시
- [ ] 인기 도시 추천 섹션 추가

## 🎯 완료 기준
- 검색 결과가 0개일 때 안내 메시지가 표시됨
- 모바일/데스크탑 모두에서 정상 동작
- 다른 빈 상태 UI와 일관성 유지

## 📝 참고 사항
- 관련 파일: `app/search/page.tsx`
- 디자인 시스템의 Empty State 컴포넌트 활용
EOF
)"
```

## 주의사항
- 이슈 제목은 명령형으로 작성 ("추가", "수정", "제거")
- 본문은 명확하고 구체적으로 작성
- 체크박스를 사용하여 작업 진행 상황 추적 가능하게 구성
- 레이블은 최대 3-4개까지만 사용
- 이슈 생성 후 반드시 생성된 URL을 사용자에게 보여주기

## 실행
위 분석을 바탕으로 GitHub 이슈를 생성하고, 생성된 이슈 URL을 반환합니다.
